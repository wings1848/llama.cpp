# llama.cpp SWLP 项目活跃状态与路线图

---

## 1. 项目当前状态与里程碑

目前 SWLP 核心引擎已在 MSVC + CUDA（Windows 10+ 环境）下打通了完整的端到端链路：
* **已打通 NGL=0 和 Partial NGL 模式**：修复了 `ggml_cgraph` 在 C/C++ 下的 ABI 内存对齐错误，彻底解决了跨编译单元的调度崩溃问题。当前已稳定支持全 CPU 调度起步，或任意固定层数驻留 GPU 时的动态层流式传输。
* **异步 PCIe 传输流水线 (v7)**：通过多 CUDA 流重叠实现了无阻塞拷贝，经测试功能一切正常。
* **实测验证**：已验证 Qwen2.5 0.5B 到 7B 系列模型在 `--ngl 0` / `--ngl 16` 配置下跑通基准测试且无崩溃风险。

详细的跑分数据、速度提升与显存对比分析，请参见：
* **[llama.cpp-swlp测试报告.md](reports/llama.cpp-swlp测试报告.md)** (显存节省与吞吐跑分数据)

## 2. 待处理项目与未来路线图

### 阶段 2 规划 (P0 - 最高优)
* **MoE 专家粒度流式缓存**：
  - **现状**：目前 MoE 专家的权重张量以层为单位进行整体迁移，未能实现 per-expert 的按需缓存。
  - **目标**：在计算图中重新捕获 expert gate 的输出，完成 per-expert 的独立 PCIe 异步加载与动态 LRU 驱逐。

### 优化集成 (P1 - 中优)
* **CUDA Graph 深度集成**：正式打通按层独立捕获/重放的 CUDA Graph 与分片调度循环的集成，彻底消除 kernel launch 延迟。
* **双缓冲机制 (Ping-Pong Buffer)**：为层数据拷贝分配两组交替交织的 GPU Buffer，完全隐藏下一层数据由于 DMA 传输产生的 CPU 阻塞。

### 架构扩展 (P2 - 低优)
* **多 GPU 滑动窗口分配**：在多卡环境中切分并同步调度滑动窗口。
* **后端无关性抽象**：将 CUDA 专有的事件和流异步机制抽象，以便后续移植到 Metal/Vulkan 后端。

---

## 3. 已知限制

* **Turing (SM 7.5) 架构限制**：部分较老的 GPU 架构对 CUDA Graph 捕获的支持存在物理缺陷，可能导致图重放失败。
* **NGL=99 模式下显存无节省**：属于设计机制。如果通过 NGL 将所有层均一次性放置在 GPU，SWLP 会探测并将其标记为固定层，从而跳过任何驱逐和迁移过程以换取最高速度。
